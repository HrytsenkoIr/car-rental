# Car Rental

## Що я реалізувала

Я зробила тестове завдання як Laravel + Vue застосунок з окремим backend API і frontend-інтерфейсом. Реалізація не обмежується одним варіантом відображення: я додала і підсумкову таблицю по автомобілях, і календарну таблицю по днях.

У застосунку є:

- підсумкова таблиця `Summary table` з кількістю free / rented / service / all днів;
- календарна таблиця `Car calendar`, де періоди оренди показані суцільними блоками по днях;
- жовті service-блоки для періодів технічного обслуговування;
- фільтр по місяцю та року;
- пошук по назві авто, номеру, кольору і типу;
- фільтр у календарі: всі авто, авто з орендами, авто з service, авто з hourly offers, повністю вільні авто;
- показ hourly offers окремою панеллю зі списком авто, дня і часу;
- маленький індикатор hourly offer у календарній клітинці;
- експорт поточного відфільтрованого списку авто в CSV;
- додаткові поля в таблиці: color, type, body type, plate, year.

## Архітектура

Проєкт реалізовано на Laravel у підході MVC з додатковими Service та Repository шарами. Controller відповідає за HTTP/API, Service містить алгоритм, Repository ізолює SQL-запити, а Models використовуються як Eloquent-мапінг основних таблиць із relations та casts.

Основні частини:

- `BookingController` приймає HTTP-запит і повертає JSON для календаря.
- `BookingService` містить основний алгоритм розрахунку вільних, зайнятих і service-днів.
- `BookingRepository` відповідає за SQL-запити до таблиць бронювань.
- `CarRepository` відповідає за SQL-запити до таблиць авто.
- Vue-компоненти відповідають тільки за інтерфейс і взаємодію з користувачем.

Такий поділ зроблено для того, щоб SQL, алгоритм і UI не були змішані в одному файлі.

## Алгоритм

Для кожного автомобіля за вибраний місяць я беру всі booking-и, які перетинаються з цим періодом.

До уваги беруться:

- тільки booking-и з `status = 1`;
- тільки авто з переданим `company_id`; якщо параметр не переданий, використовується `company_id = 1`;
- тільки авто з `status = 1`;
- тільки авто, де `is_deleted != 1` або `is_deleted IS NULL`;
- `payment_status` не використовується;
- сортування авто йде за `car_id ASC`.

Для кожного дня місяця я перевіряю проміжок з 09:00 до 21:00. День вважається вільним, якщо в цьому проміжку є 9 або більше годин підряд, коли авто не зайняте booking-ами.

Якщо booking-и перекривають день так, що немає 9 годин підряд вільного часу, день вважається зайнятим:

- якщо всі booking-и цього дня мають `source = car-service`, день позначається як `service`;
- в інших випадках день позначається як `rented`.

Для розрахунку я:

1. Беру всі booking-и, що перетинаються з днем.
2. Обрізаю їх до робочого вікна 09:00-21:00.
3. Об'єдную overlapping intervals.
4. Шукаю найбільший безперервний вільний проміжок.
5. Якщо цей проміжок `>= 9` годин, день free, інакше rented або service.

Timezone враховується через `Etc/GMT-2`, оскільки в умові вказано звернути увагу на `timezone - 2`.

## Примітка щодо звірки з еталонними даними (ТЗ)

При ручній перевірці алгоритму я знайшла розбіжності з еталонною таблицею з ТЗ для кількох
авто. Один з прикладів - id=3733 (Rolls Royce Wraith, номер U90057), 12 січня 2023:

```
3733   2023-01-12 00:00:00   2023-01-12 18:00:00   car-service
3733   2023-01-12 22:00:00   2023-01-13 22:00:00   dwm
```

За критерієм "вільне, якщо не зайняте 9+ годин підряд з 9:00 до 21:00": бронювання dwm починається о 22:00
і у вікно 9-21 цього дня не потрапляє, а car-service (00:00-18:00) залишає в межах вікна лише 18:00-21:00
вільного часу — 3 години. Це менше 9, тож день не може вважатись вільним. Однак в еталоні для цього авто 
вказано service = 0 за весь місяць, що суперечить наявності запису про сервіс, який тривав з 00:00 до 18:00
і перекривав більшу частину денного вікна. Хоча еталонні розрахунки могли базуватися на іншій логіці,
мій алгоритм реалізований чітко за критерієм із ТЗ, підтверджений успішним збігом на рекомендованому авто id=3715,
і працює без додаткових припущень.

## Дані автомобілів

Назва авто береться з `rc_cars_models_translations.name` для `lang = en`. Якщо перекладу немає, використовується `rc_cars_models.slug`, а потім fallback `car-{car_id}`.

Додатково я виводжу:

- `color` з `rc_cars_translations.attribute_color`;
- `type` з `rc_cars_models.type`;
- `body_type` через окремий `CarBodyService`;
- `registration_number` як `number`;
- `attribute_year` як `year`.

## Hourly offers

Я додала додатковий режим `Show hourly offers`. Він не змінює основний алгоритм розрахунку free/rented/service, а тільки додає окрему візуалізацію booking-ів з `source = hourly`.

Якщо режим увімкнений:

- backend додає до кожного авто масив `hourly_offers`;
- frontend показує окрему панель зі списком hourly offers;
- у календарі біля відповідного дня з'являється маленький синій індикатор.

Це зроблено окремо від основного алгоритму, щоб не ламати умову про врахування всіх booking-ів зі `status = 1`.

## Frontend

Frontend зроблено на Vue 3.

Компоненти:

- `FilterBar.vue` - вибір місяця і року;
- `CalendarTable.vue` - підсумкова таблиця записів по авто;
- `CarCalendar.vue` - календарна таблиця по днях;
- `App.vue` - завантаження даних, перемикання між таблицями, стан `showHourly`.

У календарній таблиці rental/service періоди показані не окремими клітинками, а суцільними блоками на кілька днів. Це робить періоди бронювання більш зрозумілими візуально.

## API

Основний endpoint:

```text
GET http://127.0.0.1:8000/api/calendar?year=2023&month=1
```

З hourly offers:

```text
GET http://127.0.0.1:8000/api/calendar?year=2023&month=1&show_hourly=1
```

З company id:

```text
GET http://127.0.0.1:8000/api/calendar?year=2023&month=1&company_id=1
```

Приклад структури відповіді для авто:


## Реверс-інжиніринг та мапінг типів кузова (Body Types)

У наданій базі даних таблиця автомобілів містить лише числовий ідентифікатор `car_body_id`, проте документація чи окремий довідник для розшифрування цих ID відсутні. Водночас еталонна таблиця результатів у ТЗ вимагає виведення текстових назв кузовів (`sedan`, `suv`, `convertible`, `coupe` тощо).

Для вирішення цієї проблеми я провела аналіз наявних у базі моделей та закономірностей їхнього маркування через зіставлення конкретних моделей авто, таких як `Audi R8 Spyder`, `Rolls-Royce Dawn`, `Cadillac CT6`, `Ford F-150` тощо.

На основі цього аналізу я виявила хаотичність та дублювання деяких legacy-ID в оригінальній базі, які згрупувала та замапила в коді через `CarBodyService`:

- `sedan` — об'єднує `1`, `4` та `14` (наприклад, `Mercedes S500`, `Cadillac CT6`, `Chrysler 300C`). В оригінальній базі це, скоріш за все, були різні підкласи седанів, але за ТЗ вони виводяться єдиним типом.
- `suv` — об'єднує `9` та `11` (`BMW X7`, `Land Cruiser`, `Alfa Romeo Stelvio`).
- `convertible` — об'єднує `0` та `7` (`Rolls Royce Dawn`, `Audi R8 Spyder`). ID `0` у базі часто використовується як дефолтний для преміум-кабріолетів або legacy-записів.
- `coupe` — ID `6` (`Rolls Royce Wraith`, `Ferrari 812`).
- `pickup` — ID `8` (`Ford F-150`).
- `hatchback` — ID `3` (`Alfa Romeo Giulietta`).
- `van` — об'єднує `12` та `13` (`Hyundai H1`, `VW Crafter`).

Архітектурно я винесла цей мапінг в окремий сервіс `CarBodyService`, щоб не змішувати розшифрування legacy-даних з основним алгоритмом календаря.

Для того, щоб додаток залишався стійким до можливих аномальних чи нових ID у базі, у PHP-операторі `match` передбачено безпечний дефолтний варіант:

```php
default => 'other'
```

Це гарантує, що інтерфейс не впаде з критичною помилкою при появі невідомого ID, а коректно відобразить нейтральний тип кузова.

## Тести

Я замінила стандартні Laravel example-тести на тести, які перевіряють саме реалізовану логіку тестового завдання.

Unit-тести для `BookingService` перевіряють:

- що `car-service` використовує той самий поріг 9 годин вільного часу;
- що день вважається free, якщо є 9 або більше годин підряд вільного часу у вікні 09:00-21:00;

Feature-тести для `BookingController` перевіряють:

- що endpoint `/api/calendar` повертає JSON з calendar data;
- що параметр `show_hourly=1` передається у service;
- що невалідний місяць повертає HTTP 422;
- що невалідний `company_id` повертає HTTP 422.

Запуск тестів:

```bash
php artisan test
```

Поточний результат:

```text
7 passed, 24 assertions
```
